5. **Task [Extended] - Real fix: mount the highlight StyleModule** (v1.0.2): Live Playwright DOM probe overturned the cold-chunk theory - real cause is the unmounted CodeMirror highlight StyleModule, not missing spans<br>
   **Result**: Highlighting was still broken after the retry-window and full extension-bisection rounds, so I stopped theorising and probed the live JupyterHub Markdown Preview through the headless Playwright Python API (token from env, never leaked). The token `<span>`s were ALL present and correctly classed (`ͼs`, `ͼ11`, ...) yet every one computed to `rgb(192,192,192)`, identical to the plain code colour. `hiRules: 0` across both `document.styleSheets` and `adoptedStyleSheets`, and `colorRulesMatchingSpan: []` - nothing defines those `ͼ*` classes. They come from a CodeMirror `StyleModule` (`jupyterHighlightStyle`) that is mounted only when an `EditorView` is created via `syntaxHighlighting`, which is exactly why opening the file in the editor coloured the preview and kept it coloured. Injecting `.ͼs{color:red}` live flipped the span red, proving the mechanism before writing any code. Fix: `injectHighlightRules()` in `src/highlight.ts` mounts `jupyterHighlightStyle.module.getRules()` into one `<style>` at activation; colours are `--jp-mirror-editor-*-color` vars so they track the theme. Ripped out the misaimed `MutationObserver` re-highlight plus language-chunk warming (targeted a missing-spans case never observed here): rewrote `src/index.ts` (no registry dependency), cut the jest suite to 3 tests, replaced the Galata recovery test with a StyleModule-mounted assertion, rewrote `README.md` on the sibling-fix pattern, and corrected the root-cause note in `.claude/CLAUDE.md`, the `package.json` description and CHANGELOG. Verified end-to-end on the live built extension (python keyword `rgb(186,114,186)`). Version bumped 0.6.9 → 1.0.2 at user request.
